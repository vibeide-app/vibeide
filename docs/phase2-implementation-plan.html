Pre-Phase: Spikes and Prototypes

Week 0 — 3–5 days. Before committing to the full build, validate the three highest-risk technical unknowns with isolated proof-of-concept programs.

Spike 1: wgpu + cosmic-text Glyph Atlas

Goal: Render a 120×40 grid of monospaced text at 120fps using wgpu with a glyph atlas built from cosmic-text.

What to prove:

  • cosmic-text can rasterize JetBrains Mono glyphs into a texture atlas
  • A single instanced draw call can render 4,800+ character cells
  • Frame time stays under 8.3ms (120fps) on integrated GPUs
  • Atlas updates (new glyphs, different sizes) don’t cause frame drops

Risk: cosmic-text’s glyph rasterization may have quality issues at small sizes, or the atlas-packing strategy may not be efficient enough. Mitigation: fall back to fontdue (proven in GPU terminal projects like Rio).

Spike 2: alacritty_terminal as Library

Goal: Use alacritty_terminal crate to parse VT escape sequences and maintain a terminal grid state, without the Alacritty window/renderer.

What to prove:

  • alacritty_terminal::Term can be used standalone (detached from Alacritty’s event loop)
  • PTY output can be fed through the parser and terminal grid state read back
  • Color attributes, cursor position, and scrollback are accessible
  • The crate compiles cleanly as a dependency (no hidden Alacritty-specific assumptions)

Risk: Tight coupling to Alacritty internals. Mitigation: use vte crate directly for parsing and maintain a custom grid state.

Spike 3: whisper-rs Push-to-Talk

Goal: Capture microphone audio on push-to-talk, transcribe locally with whisper-rs, and output the text.

What to prove:

  • whisper-rs works on Linux (PulseAudio/PipeWire) with acceptable latency
  • The small/base Whisper model provides usable accuracy for English developer commands
  • Transcription completes within 1–2 seconds of releasing the push-to-talk key
  • Audio capture does not block the main thread (runs on a separate tokio task)

Risk: Model loading time (~2–5s) and transcription latency. Mitigation: pre-load at startup; use “tiny” model for speed; support cloud STT fallback.

Spike Results Gate

  • All pass: proceed with the phased plan below.
  • Spike 1 fails: evaluate fontdue or OpenGL glyph rendering.
  • Spike 2 fails: plan for custom VT parser using vte crate + custom grid.
  • Spike 3 fails: defer voice to Phase 2.7+ and ship with cloud-only STT initially.

Phase 2.1: Foundation

Window, GPU Context, Event Loop — 2 weeks

Goal

Create the application skeleton: a native window powered by winit, a wgpu rendering context, and an async event loop that can process input events and schedule frame redraws at 120fps.

Key Tasks

  1. Cargo workspace setup — Create a multi-crate workspace with clear boundaries:
    • vibeide-app — binary crate, owns the event loop and window
    • vibeide-gpu — wgpu device/surface management, render pipeline abstraction
    • vibeide-terminal — terminal emulation and PTY
    • vibeide-ui — UI chrome, layout engine
    • vibeide-agent — agent process management
    • vibeide-voice — audio capture and STT
    • vibeide-common — shared types, config, error types
  2. winit window creation — Open a resizable window with title “VibeIDE”, minimum size 800×600, start maximized. Handle DPI scaling from the start.
  3. wgpu initialization — Request adapter (prefer discrete GPU, fall back to integrated), create device and queue. Create a surface for the window. Configure swap chain for Mailbox present mode with sRGB format.
  4. Render loop — Implement a frame-driven loop: poll winit events, process input, call render, present. Target 120fps with frame pacing.
  5. Clear screen test — Render a solid dark background color (#0f172a) to confirm the full pipeline works.
  6. Input event scaffolding — Wire keyboard and mouse events from winit into an internal event channel. No processing yet, just logging.
  7. Configuration system — Load app config from ~/.vibeide/config.toml using the toml crate. Define config structs for: window size/position, font settings, theme colors, keybindings.
  8. Error handling foundation — Define VibeError enum using thiserror. Set up tracing with tracing-subscriber for structured logging.

Risk Areas

  • winit + wgpu compatibility: winit 0.30 changed the event loop API significantly. Pin to a known-good version pair and test on both Wayland and X11.
  • Wayland vs X11: Test both display servers on Linux from day one. Verify window decoration behavior on both.
  • DPI scaling edge cases: High-DPI and mixed-DPI monitors need testing. Store logical vs physical size correctly from the start.

What Can Be Ported from Phase 1

  • Theme colors and design tokens — CSS variables map directly to wgpu clear colors and UI rendering colors.
  • Configuration schema — Settings structure (font size, theme, keybindings) can be reused; just change storage format to plain TOML.
  • Keybinding definitions — Keybinding map can be ported as-is; only the event source changes.

Testing Strategy

  • Unit tests: Config loading with valid/invalid TOML files. Error type formatting. Event channel send/receive.
  • Integration tests: Window creation succeeds on headless CI. Config file round-trip.
  • Manual verification: Window opens, shows dark background, responds to resize, logs input events.

Definition of Done

  • cargo run opens a native window with dark background
  • Window resizes smoothly without flickering
  • Frame time logged, consistently under 2ms for empty frame
  • Input events (keyboard, mouse) logged to terminal
  • Config loaded from ~/.vibeide/config.toml or defaults
  • tracing output to both file and stderr
  • All crates in workspace compile with zero warnings
  • CI passes with cargo clippy and cargo test

Phase 2.2: Terminal Rendering

GPU Text and PTY — 3 weeks

Goal

Render a fully functional terminal in the GPU window: a PTY-connected shell with proper VT100/VT220 escape sequence handling, a GPU-rendered glyph atlas for text, and cursor rendering.

Key Tasks

  1. Glyph atlas construction — Use cosmic-text to rasterize the configured monospace font (default: JetBrains Mono) into a texture atlas. Pack glyphs into a 2048×2048 GPU texture. Record UV coordinates and metrics. Support dynamic atlas growth for CJK and emoji.
  2. Cell rendering shader — Write a WGSL vertex/fragment shader pair for instanced rendering of terminal cells. Each cell instance: position, glyph UV rect, foreground color, background color, attributes (bold, italic, underline). Single instanced draw call for the entire grid.
  3. PTY management — Use portable-pty crate to spawn a shell process. Wire stdout/stderr to a reader task, stdin to a writer task. Handle PTY resize (SIGWINCH).
  4. VT parser integration — Feed PTY output through alacritty_terminal::Term to parse escape sequences and maintain the terminal grid state.
  5. Cursor rendering — Blinking block/beam/underline cursor (configurable). Blink rate configurable (default: 530ms). Shape changes based on terminal mode.
  6. Scrollback buffer — 10,000 lines default (configurable). Smooth scrolling with mouse wheel and keyboard (Shift+PageUp/Down).
  7. Selection and copy — Mouse drag text selection with semi-transparent overlay. Clipboard copy via arboard crate. Word-select (double-click) and line-select (triple-click).
  8. Color scheme — ANSI 16-color, 256-color, and 24-bit true color support. OSC 4 color queries and changes.
  9. Performance optimization — Dirty rect tracking for changed cells only. Batch PTY reads. Target <4ms frame time.

Risk Areas

  • Glyph atlas overflow: Emoji and CJK characters can balloon the atlas. Implement atlas paging or LRU eviction.
  • alacritty_terminal API stability: Not designed as a public library; pin exact version and wrap behind an abstraction trait.
  • PTY read performance: High-throughput output must not drop frames. Buffer reads and process in batches.
  • Unicode edge cases: Combining characters, RTL text, and variation selectors need careful handling.

What Can Be Ported from Phase 1

  • Terminal color theme — xterm.js theme configuration maps directly to the new color palette.
  • Terminal settings — Font family, font size, cursor style, scrollback size carry over.
  • Selection behavior logic — Double-click/triple-click semantics can be reused conceptually.

Testing Strategy

  • Unit tests: Glyph atlas packing, color mapping, cell attribute parsing.
  • Integration tests: Known VT sequences through parser and verify grid state. PTY spawn and I/O. Scrollback correctness.
  • Visual regression tests: Render known state to image and compare. Use vttest for compliance.
  • Performance tests: Frame time with full grid. PTY throughput with cat /dev/urandom.

Definition of Done

  • Full terminal emulator rendering in GPU window
  • Shell prompt appears and accepts input
  • Colors (16, 256, true color) render correctly
  • Cursor blinks and changes shape
  • Scrollback works with mouse wheel and keyboard
  • Text selection and clipboard copy functional
  • vttest basic tests pass
  • Frame time <4ms for single terminal pane
  • PTY throughput >100MB/s

Phase 2.3: Multi-Terminal

Splits, Tabs, and Layout — 2 weeks

Goal

Support multiple terminal panes in a tiled layout with splits (horizontal/vertical), tabs, and keyboard-driven navigation. Each pane runs an independent PTY session.

Key Tasks

  1. Layout tree data structure — Tree structure for pane layout: leaf nodes are terminal panes, internal nodes are splits with configurable ratio. Support arbitrary nesting.
  2. Split operations — Split-horizontal (Ctrl+Shift+H) and split-vertical (Ctrl+Shift+V). New split creates a new PTY session. Close pane (Ctrl+Shift+W) with tree rebalancing.
  3. Pane focus and navigation — Navigate with Ctrl+Shift+Arrow or Ctrl+Shift+H/J/K/L (vim-style). Accent-colored border on focused pane.
  4. Pane resizing — Drag divider to resize. Keyboard resize with Ctrl+Shift+Alt+Arrow. Minimum: 20 columns × 5 rows. Double-click divider to equalize.
  5. Tab support — Each workspace is a tab with a layout tree. Ctrl+Shift+T (new tab), Ctrl+Tab / Ctrl+Shift+Tab (cycle). Tab bar rendered at the top.
  6. Layout serialization — Serialize layout (split ratios, tabs, working directories) to ~/.vibeide/layout.json for restore on launch.
  7. Viewport and rendering — Each pane renders into a scissored viewport. Dividers as 1px lines in theme border color.
  8. PTY resize propagation — Correct SIGWINCH on pane resize so the shell reflows correctly.

Crates / Dependencies

CratePurpose
serde_jsonLayout serialization
(no new crates)Layout logic is custom

Risk Areas

  • Layout tree complexity: Deeply nested splits can create tiny panes. Enforce minimum size constraints at the tree level.
  • PTY resize timing: Debounce resize events (16ms) to avoid shell reflow thrashing.
  • Rendering performance: With 6+ panes, verify frame time stays under 8ms.

What Can Be Ported from Phase 1

  • Split logic and keybindings — Tree data structure logic is similar but needs Rust reimplementation.
  • Tab management state — Tab ordering, active tab tracking, and naming carry over conceptually.
  • Layout persistence — Save/restore approach from Phase 1 can be adapted.

Testing Strategy

  • Unit tests: Layout tree operations, minimum size enforcement, serialization round-trip, focus traversal.
  • Integration tests: Multiple independent PTYs. Resize propagation correctness.
  • Manual testing: Rapid split/resize/close. Verify no orphaned PTY processes.

Definition of Done

  • Horizontal and vertical splits working
  • Keyboard navigation between panes (arrow and vim-style)
  • Pane resize by dragging divider
  • Tab bar with create/close/switch
  • Each pane has an independent shell session
  • Layout persists across app restarts
  • No orphaned PTY processes after closing panes
  • Frame time <8ms with 4 panes

Phase 2.4: UI Chrome

Command Palette, Status Bar, Menus — 3 weeks

Goal

Build the non-terminal UI elements: a command palette for quick actions, a status bar for system information, and modal dialogs. All rendered natively with the GPU — no HTML, no webview.

Key Tasks

  1. Immediate-mode UI primitives — GPU-rendered rectangles, text labels, text input fields, scrollable lists, buttons. Shared glyph atlas with terminal rendering.
  2. Command palette — Fuzzy-search palette (Ctrl+Shift+P). Text input, filtered command list, keyboard navigation, skim fuzzy matching, action dispatch. 100ms slide-down animation.
  3. Status bar — Bottom bar. Left: CWD, git branch, active agent status. Right: CPU/memory usage, terminal dimensions, notification count.
  4. Quick Terminal selector — Dropdown (Ctrl+Shift+J) listing all open terminal sessions with CWD, recent command, agent assignment.
  5. Settings UI — Modal panel for editing settings with live preview. Saves to ~/.vibeide/config.toml.
  6. Keyboard shortcut overlay — Help overlay (Ctrl+?) with all keybindings grouped by category. Searchable.
  7. Theme engine — Load theme from config. Built-in themes (dark, light, high-contrast). Instant switching.
  8. Modal system — Generic modal dialogs for confirmations, input prompts, and errors with semi-transparent overlay.
  9. Animations — Smooth transitions for palette open/close, focus change, tab switch. All under 150ms.

Phase 2.5: Agent Orchestration

AI Process Management — 2 weeks (parallel with 2.4)

Goal

Manage AI agent processes (Claude Code CLI, Codex CLI, or any MCP-compatible agent) as first-class terminal sessions. Spawn agents in dedicated PTY sessions, track their lifecycle, and provide agent-aware UI.

Key Tasks

  1. Agent process abstraction — Define an Agent trait: spawn(config) -> AgentHandle, send_input(text), read_output() -> Stream<String>, terminate(), status() -> AgentStatus. Implement for Claude Code CLI and Codex CLI.
  2. Agent session management — AgentManager tracks all running agents. Each agent in its own PTY. Start, stop, restart. Concurrent limit (default: 4).
  3. Agent configuration — Config per agent: name, command, env vars, working directory, auto-restart policy. Stored in ~/.vibeide/agents.toml. Project-local support via .vibeide/agents.toml.
  4. Agent-aware pane decoration — Colored border per agent type, agent name in tab title, status indicator (running/thinking/idle/error) in status bar.
  5. Agent input routing — Keyboard input to agent PTY stdin. Interrupt (Ctrl+C), kill (force-terminate), and restart commands.
  6. Agent output processing — Parse output for thinking indicators, tool use, code blocks, errors. Tag output sections with metadata.
  7. Agent health monitoring — Detect unexpected exit. Auto-restart if configured. Notification on crash. Log all lifecycle events.
  8. Multi-agent coordination — Multiple simultaneous agents in separate panes with independent contexts. No inter-agent communication in this phase.

Crates / Dependencies

CratePurpose
tokio::processAgent process spawning and management
notifyFile system watching for agent config changes

Risk Areas

  • Agent process lifecycle: Agents may hang, crash, or produce unexpected output. Implement robust timeouts and health checks.
  • PTY interaction with agents: Some agents use interactive terminal features (spinners, progress bars). Terminal emulator must handle correctly.
  • Resource management: Monitor per-agent resource usage and warn when approaching limits.

What Can Be Ported from Phase 1

  • Agent spawn logic — Process spawning and I/O routing from Phase 1’s agent manager.
  • Agent configuration schema — Agent definition format carries over.
  • Agent status tracking — Lifecycle state machine can be reused.

Testing Strategy

  • Unit tests: AgentManager lifecycle, config parsing, status state machine transitions.
  • Integration tests: Mock agent (echo script) I/O routing. Crash and auto-restart. Concurrent limit enforcement.
  • E2E tests: Real Claude Code instance (if API key available), simple prompt and response verification.

Definition of Done

  • Agents can be spawned in dedicated terminal panes
  • Agent panes have visual indicators (border color, status)
  • Agent lifecycle (start/stop/restart) works reliably
  • Unexpected exit triggers notification and optional auto-restart
  • Multiple agents can run simultaneously
  • Agent config loaded from TOML files
  • Memory usage stays under 150MB with 4 agents running
  • All agent lifecycle events logged

Phase 2.7: Voice Integration

Push-to-Talk and Transcription — 2 weeks (parallel with 2.6)

Goal

Add voice input capability: push-to-talk to capture audio, local transcription using Whisper, and input of the transcribed text into the focused terminal or agent pane.

Key Tasks

  1. Audio capture — cpal crate for cross-platform audio input at 16kHz mono. Dedicated capture thread.
  2. Push-to-talk activation — Configurable key (default: Right Alt or F13). Visual indicator (microphone icon, pulsing accent color) while recording.
  3. Local transcription with whisper-rs — Load model at startup in background. Transcribe on separate tokio task. Configurable model size (tiny/base/small/medium).
  4. Cloud STT fallback — OpenAI Whisper API, Deepgram support. Configured with API key in settings. Used when local model disabled or unavailable.
  5. Transcription output — Inject text into focused pane’s PTY stdin. Optional preview overlay for editing before injection. Voice command mapping (“run tests” → actual commands).
  6. Voice activity detection — Energy-based VAD to trim silence. Avoid sending empty audio to transcriber.
  7. Audio feedback — Subtle sound or visual flash on recording start/stop. Configurable: sounds on, sounds off, visual only.
  8. Privacy controls — All voice data processed locally by default. No cloud unless explicitly configured. Config option to disable voice entirely. No audio stored after transcription.

  • Latency budget: Capture + transcription must complete in <2s. Tiny model is faster but less accurate.

  • Entirely new. Phase 1 did not include voice. However, PTY input injection mechanism reuses Phase 2.2/2.3 infrastructure.

Definition of Done

  • Push-to-talk captures audio and transcribes with Whisper
  • Visual indicator shows when recording
  • Transcribed text injected into focused terminal pane
  • Cloud STT fallback configurable
  • Transcription completes in <2 seconds
  • Voice can be fully disabled in settings
  • No audio data stored or sent to cloud by default
  • Works with PulseAudio and PipeWire on Linux

  • Update tests: Mock server for update check simulation. End-to-end update flow.

  • Platform-specific keyboard shortcuts work correctly
  • Packaging produces installable artifacts for all platforms
  • CI builds and tests on all three platforms
  • Auto-update check and notification works
  • No platform-specific crashes or rendering issues

Phase 2.10: WASM Plugin System

Stretch Goal — 2–3 weeks

Goal

Enable extensibility through a WASM-based plugin system. Plugins run in a sandboxed WASM runtime and can extend VibeIDE with custom commands, UI panels, themes, and agent integrations.

Key Tasks

  1. WASM runtime integration — Embed wasmtime as the plugin runtime. Each plugin in its own sandboxed instance with limited host access.
  2. Plugin API definition — Stable plugin API via wit-bindgen: register commands, add status bar items, create UI panels, listen to events. Version the API.
  3. Plugin manifest — plugin.toml manifest: name, version, author, permissions (filesystem, network, terminal). User must approve permissions on install.
  4. Plugin marketplace — Built-in browser from command palette. GitHub-based registry. Install, update, uninstall.
  5. Built-in plugins — First-party examples: theme switcher, git integration panel, markdown preview. Serve as developer examples.
  6. Plugin development kit — vibeide-plugin-sdk crate with template project, documentation, and hot-reload development workflow.

Key Architectural Decisions

1. No Webview

All rendering is native GPU via wgpu. This eliminates the Chromium memory overhead (~200–400MB) and gives full control over the render pipeline.

2. alacritty_terminal for VT Parsing

Rather than writing a VT parser from scratch, we leverage the battle-tested parser from Alacritty. This saves months of work on escape sequence edge cases.

3. Cargo Workspace with Small Crates

Each subsystem (GPU, terminal, UI, agent, voice) is a separate crate with clear API boundaries. This enforces separation of concerns and enables parallel compilation.

4. Immediate-Mode UI for Chrome

UI elements (command palette, status bar, modals) use immediate-mode rendering into the same GPU pipeline. No retained-mode widget tree — keeps the rendering architecture simple and fast.

5. WASM for Plugins

Plugins run in a sandboxed WASM runtime, not native code. This provides security guarantees and cross-platform compatibility at the cost of some performance overhead (acceptable for plugin-level operations).

6. Local-First Voice

Voice transcription runs locally with Whisper by default. Cloud STT is opt-in. No audio leaves the device unless the user explicitly configures cloud transcription.
